# 评估清单

## 一、开发清单评估标准

### 1. 需求覆盖度（权重：30%）
- [ ] 是否覆盖所有功能需求
- [ ] 是否覆盖非功能需求（性能、安全、可维护性）
- [ ] 是否识别技术约束
- [ ] 验收标准是否明确可测

### 2. 任务拆分合理性（权重：25%）
- [ ] 功能点粒度是否合适（建议：1-3 小时工作量）
- [ ] 功能点是否单一职责
- [ ] 功能点之间是否低耦合
- [ ] 是否遗漏必要的准备工作（环境搭建、依赖安装等）

### 3. 依赖关系（权重：20%）
- [ ] 功能点顺序是否合理
- [ ] 依赖关系是否清晰标注
- [ ] 是否存在循环依赖
- [ ] 是否有并行开发的可能性

### 4. 技术可行性（权重：15%）
- [ ] 技术选型是否合理
- [ ] 实现方案是否可行
- [ ] 是否考虑现有代码结构
- [ ] 是否有技术风险点

### 5. 风险识别（权重：10%）
- [ ] 是否识别潜在风险
- [ ] 是否有应对方案
- [ ] 是否有回退策略
- [ ] 是否考虑边界情况

---

## 二、功能点实现评估标准

### 1. 功能正确性（权重：35%）
- [ ] 核心功能是否按需求实现
- [ ] 边界条件是否处理
- [ ] 错误情况是否处理
- [ ] 是否满足验收标准

### 2. 代码质量（权重：30%）

#### 2.1 可读性
- [ ] 命名是否清晰有意义
- [ ] 代码结构是否合理
- [ ] 注释是否充分且必要
- [ ] 函数/方法长度是否适中

#### 2.2 可维护性
- [ ] 是否遵循 DRY 原则
- [ ] 是否遵循 SOLID 原则
- [ ] 是否有适当的抽象
- [ ] 是否易于测试

#### 2.3 性能
- [ ] 是否有明显性能问题
- [ ] 是否有不必要的计算
- [ ] 是否有内存泄漏风险
- [ ] 是否有优化空间

#### 2.4 安全性
- [ ] 是否有 SQL 注入风险
- [ ] 是否有 XSS 风险
- [ ] 是否有敏感信息泄露
- [ ] 是否有权限控制

### 3. 项目一致性（权重：20%）
- [ ] 是否遵循现有代码风格
- [ ] 是否使用项目现有的工具库
- [ ] 是否遵循项目架构模式
- [ ] 是否符合项目命名规范

### 4. 测试覆盖（权重：15%）
- [ ] 是否有必要的单元测试
- [ ] 测试用例是否覆盖主要场景
- [ ] 测试用例是否覆盖边界条件
- [ ] 测试是否可重复运行

## 三、严重程度定义

### 高（必须修复）
- 功能无法正常工作
- 存在安全漏洞
- 存在数据丢失风险
- 严重性能问题
- 违反核心需求

### 中（应该修复）
- 代码质量明显不佳
- 缺少必要的错误处理
- 命名不清晰
- 缺少必要的注释
- 不符合项目规范

### 低（建议修复）
- 可优化的代码结构
- 可改进的命名
- 可补充的注释
- 可提升的性能

---

## 四、评估通过标准

### 清单评估通过条件
- 需求覆盖度 ≥ 90%
- 无"高"严重程度问题
- "中"严重程度问题 ≤ 3 个

### 功能点评估通过条件
- 功能正确性 100% 通过
- 无"高"严重程度问题
- "中"严重程度问题 ≤ 2 个
- 代码质量评分 ≥ 80 分
