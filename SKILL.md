# Coder + Evaluator 循环开发工作流

这是一个双层循环的开发工作流，确保代码质量通过迭代式开发和评估。

## 架构概述

```
用户需求
    ↓
[Master 协调器]
    ↓
┌─────────────────────────────────────┐
│ 第一层循环：开发清单评估              │
│                                     │
│  Coder 列出开发清单                  │
│         ↓                           │
│  Evaluator 评估清单                  │
│         ↓                           │
│  [通过?] → 否 → Coder 修改清单      │
│         ↓                           │
│        是                           │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 第二层循环：功能点开发评估（对每个功能点）│
│                                     │
│  Coder 实现功能点                    │
│         ↓                           │
│  Evaluator 评估实现                  │
│         ↓                           │
│  [通过?] → 否 → Coder 修改实现      │
│         ↓                           │
│        是                           │
│         ↓                           │
│  下一个功能点                        │
└─────────────────────────────────────┘
    ↓
完成所有功能点
```

## 工作流详细步骤

### 阶段 1：需求分析与清单生成

**Coder 职责：**
1. 分析用户需求，识别：
   - 功能需求（必须实现什么）
   - 非功能需求（性能、安全、可维护性）
   - 技术约束（框架、语言、依赖）
   - 验收标准（如何判断完成）

2. 生成开发清单，格式如下：
   ```markdown
   # 开发清单

   ## 功能点 1: [名称]
   - 描述: [详细说明]
   - 涉及文件: [文件列表]
   - 预计复杂度: [高/中/低]
   - 依赖: [依赖的其他功能点或外部库]

   ## 功能点 2: [名称]
   ...

   ## 开发顺序建议
   1. [功能点 X] - 理由
   2. [功能点 Y] - 理由
   ...
   ```

**Evaluator 评估清单标准：**
- 需求覆盖度：清单是否覆盖所有用户需求
- 任务拆分合理性：功能点粒度是否合适（不过大也不过小）
- 依赖关系：功能点顺序是否合理，依赖是否清晰
- 技术可行性：实现方案是否可行
- 风险识别：是否遗漏潜在风险点

### 阶段 2：清单评估循环

```
循环开始（最多 3 次）：
  Coder 提交清单
       ↓
  Evaluator 评估
       ↓
  评估结果：
    - 通过 → 进入阶段 3
    - 部分问题 → 返回具体反馈，Coder 修改后重新提交
    - 严重问题 → 重新分析需求，从头生成清单
循环结束
```

**Evaluator 反馈格式：**
```markdown
# 清单评估报告

## 总体评价
[通过/需要修改/需要重新设计]

## 问题列表
1. [严重程度: 高/中/低] [问题描述]
   - 影响: [会如何影响开发]
   - 建议: [如何修改]

2. ...

## 优点
- [清单中做得好的地方]

## 建议
- [整体改进建议]
```

### 阶段 3：功能点开发循环

对清单中的每个功能点，执行以下循环：

```
功能点循环开始：
  Coder 实现当前功能点
       ↓
  Evaluator 评估实现
       ↓
  评估结果：
    - 通过 → 标记完成，进入下一个功能点
    - 问题 → 返回具体反馈，Coder 修改后重新评估
       ↓
  循环直到通过（最多 3 次）
功能点循环结束
```

**Coder 实现要求：**
1. 遵循项目现有代码风格和规范
2. 添加必要的注释和文档
3. 处理错误情况和边界条件
4. 编写或更新相关测试（如果项目有测试要求）
5. 更新相关文档

**Evaluator 功能点评估标准：**
- 功能正确性：是否按需求实现
- 代码质量：
  - 可读性：命名、注释、结构是否清晰
  - 可维护性：是否遵循 SOLID 原则
  - 性能：是否有明显性能问题
  - 安全性：是否有安全漏洞
- 需求覆盖度：是否满足验收标准
- 项目一致性：是否与现有代码风格一致

**Evaluator 功能点评估报告格式：**
```markdown
# 功能点评估报告: [功能点名称]

## 实现状态
[通过/需要修改]

## 检查结果

### 功能正确性
- [✓/✗] 核心功能实现
- [✓/✗] 边界条件处理
- [✓/✗] 错误处理

### 代码质量
- [✓/✗] 命名规范
- [✓/✗] 代码结构
- [✓/✗] 注释完整性
- [✓/✗] 性能考虑

### 需求覆盖
- [✓/✗] 验收标准 1: [描述]
- [✓/✗] 验收标准 2: [描述]

## 问题列表
1. [严重程度] [问题描述]
   - 位置: [文件:行号]
   - 建议: [如何修改]

## 改进建议
- [可选的改进点]

## 测试建议
- [建议的测试用例]
```

### 阶段 4：完成与总结

所有功能点通过评估后：

1. **Coder 生成完成报告**：
   ```markdown
   # 开发完成报告

   ## 功能清单
   - [✓] 功能点 1: [名称]
   - [✓] 功能点 2: [名称]
   ...

   ## 修改的文件
   - [文件 1]: [修改说明]
   - [文件 2]: [修改说明]
   ...

   ## 新增的文件
   - [文件 1]: [用途说明]
   ...

   ## 测试情况
   [测试覆盖情况]

   ## 已知限制
   [如果有]

   ## 后续建议
   [可选的改进或扩展]
   ```

2. **Evaluator 最终确认**：
   - 检查所有功能点是否完成
   - 确认无遗漏问题
   - 提供最终质量评价

## 协调器（Master）职责

协调器负责：
1. 控制整体流程
2. 在 Coder 和 Evaluator 之间传递信息
3. 管理循环计数（防止无限循环）
4. 在循环达到上限时做出最终决策
5. 维护项目状态和进度

## 使用指南

### 何时使用此 skill
- 开发新功能或模块
- 重构现有代码
- 修复复杂 bug（涉及多文件修改）
- 任何需要结构化开发流程的任务

### 何时不使用此 skill
- 简单的单文件修改
- 明确的单行 bug 修复
- 仅文档更新
- 简单的配置更改

### 自定义选项
- 可以调整最大循环次数（默认 3 次）
- 可以启用/禁用测试要求
- 可以自定义评估标准
- 可以调整功能点粒度

## 配置参数

在 skill 目录下创建 `config.json` 进行自定义：

```json
{
  "max_review_cycles": 3,
  "require_tests": true,
  "evaluation_strictness": "medium",
  "auto_fix_minor_issues": true
}
```
